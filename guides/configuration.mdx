# Configuration

InsForge uses environment variables for all configuration. No config files, no UI settings, just environment variables.

## Core Settings

```bash
# Required
DATABASE_URL=postgresql://user:pass@localhost:5432/insforge
BETTER_AUTH_SECRET=your-secret-key-minimum-32-chars

# Optional
API_PORT=7130
LOG_LEVEL=info
```

## OAuth Providers

```bash
# Google
GOOGLE_CLIENT_ID=your-client-id
GOOGLE_CLIENT_SECRET=your-client-secret

# GitHub  
GITHUB_CLIENT_ID=your-client-id
GITHUB_CLIENT_SECRET=your-client-secret
```

## Storage

```bash
# Local (default)
STORAGE_BACKEND=local
UPLOAD_DIR=./uploads

# S3
STORAGE_BACKEND=s3
AWS_ACCESS_KEY_ID=your-key
AWS_SECRET_ACCESS_KEY=your-secret
AWS_REGION=us-east-1
S3_BUCKET=your-bucket
```

## Limits

```bash
MAX_FILE_SIZE=10485760  # 10MB in bytes
MAX_FILES_PER_FIELD=10  # Multiple file uploads
REQUEST_BODY_LIMIT=104857600  # 100MB
```

## Security

```bash
# CORS
CORS_ORIGIN=https://your-frontend.com

# Rate limiting
RATE_LIMIT_WINDOW=900000  # 15 minutes in ms
RATE_LIMIT_MAX=1000  # Requests per window

# Trusted origins for OAuth
TRUSTED_ORIGINS=https://your-app.com,https://staging.your-app.com
```

## Database Pool

```bash
# Connection pool settings
DB_POOL_MAX=20
DB_POOL_IDLE_TIMEOUT=30000
DB_POOL_CONNECTION_TIMEOUT=2000
```

## Development vs Production

The same Docker image runs in both environments. Only environment variables change:

**Development**: Relaxed CORS, verbose logging, local storage
**Production**: Strict CORS, error logging only, S3 storage

This design means no surprises when deploying. What works locally works in production.

## Secrets Management

Never commit secrets to Git. Use:

- `.env` files (git-ignored)
- Docker secrets
- Kubernetes secrets  
- AWS Secrets Manager
- Environment variables from your platform
